# Daily Focus Operating OS Spec

## 1. 목적

`daily-focus`를 단순 작업 메모장이 아니라,  
`오늘 해야 할 일`, `지금 막힌 일`, `다음에 무엇을 결정해야 하는지`를 자동으로 보여주는 개인 운영 OS로 확장한다.

이 문서는 실제 구현 기준 명세서이며, 앞으로 기능을 추가할 때 우선순위와 완료 기준으로 사용한다.

## 2. 제품 정의

제품명:
- `Daily Focus CC / VIEW`

제품 역할:
- 이 PC에서 동작하는 Claude Code 세션과 git 활동을 모아서
- 오늘의 작업 흐름을 자동으로 갱신하고
- 정체 작업을 감지하고
- 다음 액션을 제안하는 운영 대시보드

핵심 가치:
- 버튼을 적게 눌러도 최신 상태가 유지될 것
- “뭘 해야 하지?”보다 “이걸 지금 해야 한다”가 바로 보일 것
- 진행률보다 운영 판단 속도를 높일 것

## 3. 현재 구현 상태

이미 구현된 기능:
- 오늘 작업 목록
- 프로젝트별 CC / VIEW 버튼
- Claude 사용량 추적
- 세션 활동 피드
- 수동 체크인 + 자동 활동 기록
- 자동 새로고침
- 정체 작업 감지
- 완료 기록 기반 학습형 시간 예측
- 다음 목적 / 추천 순서 제안
- 백그라운드 실행 / 부팅 시 자동 시작
- 운영 브리프 자동 생성

## 4. 목표 사용자

1. 대표 시점
- 지금 전체 작업 흐름이 건강한지 빠르게 보고 싶다.

2. 팀장 시점
- 어떤 프로젝트가 멈췄는지, 누가 무엇을 먼저 정리해야 하는지 보고 싶다.

3. 실무자 시점
- 지금 가장 중요한 일 1개와 정리해야 할 일 1개를 바로 알고 싶다.

현재 `daily-focus`는 우선 `실무자 + 최종 브레인` 관점에 최적화한다.

## 5. 핵심 문제

기존 문제:
- 작업이 많아질수록 무엇부터 해야 하는지 판단이 느려진다.
- 오래 멈춘 작업이 눈에 띄지 않는다.
- 체크인은 남겨도 다음 액션으로 연결되지 않는다.
- 브라우저를 새로고침하거나 버튼을 눌러야 최신 상태를 보는 경우가 많다.

해결 방향:
- 자동 감시
- 정체 감지
- 다음 액션 추천
- 하루/주간 운영 요약

## 6. 기능 구조

### 6.1 자동 감시

설명:
- 서버와 대시보드는 일정 주기로 자동으로 갱신된다.
- 화면이 다시 활성화되면 즉시 새로고침한다.

완료 기준:
- 사용자가 아무 버튼도 누르지 않아도 최신 상태가 유지된다.
- 탭을 오래 두었다가 돌아와도 즉시 최신화된다.

### 6.2 정체 감지

설명:
- 작업별 예상 시간과 최근 활동 시간을 기준으로 `watch`, `stale`, `critical` 상태를 계산한다.
- 오래 멈춘 작업은 반짝이는 시각 표시로 강조한다.

완료 기준:
- 실제로 오래 멈춘 작업만 경고된다.
- 막 시작한 작업이 과도하게 빨간색이 되지 않는다.

### 6.3 운영 브리프

설명:
- 오늘 데이터에서 자동으로 다음 4가지를 계산한다.
- 헤드라인
- 오늘의 목적
- 현재 집중 포인트
- 다음 액션 3개
- 추천 작업 순서
- 리스크 / 완료 흐름

완료 기준:
- 사용자는 카드 1개만 보고도 “지금 뭘 해야 하는지” 이해할 수 있다.
- 브리프는 작업/세션 상태가 바뀌면 자동 갱신된다.

### 6.4 체크인 기반 운영

설명:
- 체크인은 단순 기록이 아니라 다음 판단을 돕는 운영 메모다.
- 최근 체크인이 없으면 브리프에서 체크인 작성을 권한다.

완료 기준:
- 체크인이 비어 있을 때 시스템이 이를 알려준다.
- 최근 체크인 문구가 운영 브리프에 요약 노트로 반영된다.

### 6.5 백그라운드 유지

설명:
- 컴퓨터 부팅 후에도 `daily-focus`가 자동 실행되고, 죽으면 다시 살아나야 한다.

완료 기준:
- 시작프로그램 기준 자동 시작
- 서버 비정상 종료 시 keepalive가 재실행
- 중복 실행 방지

### 6.6 학습형 시간 예측

설명:
- 완료된 작업의 실제 소요 시간을 샘플로 쌓고, 이후 비슷한 작업의 예상 시간을 보정한다.
- 프로젝트 + 작업 유형 + 전체 평균을 섞어서 추정한다.

완료 기준:
- 새 작업의 예상 시간이 점점 사용자의 실제 Claude Code 작업 속도에 가까워진다.
- 예측은 휴리스틱 단독보다 안정적으로 보정된다.

## 7. 다음 우선순위 기능

### P1. 정체 작업 액션화

목표:
- 경고를 보여주는 데서 끝나지 않고, 사용자가 바로 결정을 내리게 만든다.

추가 기능:
- `계속 진행`
- `보류`
- `분해`
- `내일로 이월`

완료 기준:
- `critical/stale` 작업은 반드시 액션 하나로 정리할 수 있어야 한다.

### P2. 하루 마감 리포트

목표:
- 오늘의 결과를 자동으로 정리해 내일 시작 비용을 낮춘다.

포함 항목:
- 오늘 완료한 일
- 오래 멈춘 일
- 내일 첫 작업 3개
- 범위를 줄여야 하는 프로젝트

완료 기준:
- 버튼 한 번 또는 자동 스케줄로 일일 리포트가 생성된다.

### P3. 주간 운영판

목표:
- 오늘 단위에서 끝나지 않고 주간 리듬을 관리한다.

포함 항목:
- 이번 주 목표
- 완료율
- 누적 정체 건수
- 주간 리스크 프로젝트

### P4. 프로젝트 건강도 점수

목표:
- 프로젝트별 상태를 `Healthy / Watch / Risk / Critical`로 한눈에 본다.

평가 요소:
- 최근 세션
- 최근 커밋
- 체크인 유무
- 진행률
- 마감 임박 여부

### P5. 조직형 연동

목표:
- 이후 `HIVEMEDIA Task`와 연결해 대표/팀장/실무자가 같은 흐름을 보게 한다.

연동 방향:
- 대표가 지시
- 팀장이 분담
- 실무자는 daily-focus에서 실행
- 정체는 조직 대시보드에 상향 반영

## 8. API 기준

현재 핵심 API:
- `GET /api/today`
- `GET /api/ops-brief`
- `GET /api/week`
- `GET /api/checkins`
- `POST /api/checkin`
- `POST /api/agent-status`

향후 추가 예정 API:
- `POST /api/tasks/{id}/decision`
- `GET /api/day-report`
- `GET /api/project-health`

## 9. 데이터 기준

현재 사용 데이터:
- `tasks`
- `checkins`
- `agent_status`
- `agent_activity`
- `weekly_goals`

향후 추가 권장:
- `task_decisions`
- `day_reports`
- `project_health_snapshots`

## 10. 운영 원칙

원칙 1:
- 새 작업을 늘리기보다 현재 흐름을 끝내는 쪽을 우선한다.

원칙 2:
- `critical`은 반드시 결정 대상이다. 방치 대상이 아니다.

원칙 3:
- 체크인은 회고가 아니라 다음 판단을 돕는 짧은 운영 메모다.

원칙 4:
- 시스템은 사용자의 클릭을 기다리기보다 먼저 알려줘야 한다.
