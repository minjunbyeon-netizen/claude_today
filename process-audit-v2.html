근본 원인 발견 및 수정 완료

원인: .venv/Scripts/python.exe는 전체 인터프리터가 아닌 런처 프록시(pyvenv.cfg의 executable 설정). PM2가 런처(PID A)를 시작하면, 런처가 시스템 Python(PID B)을 자식으로 10ms 내 생성. PM2는 A만 관리 → A 죽이면 B는 고아 → B가 포트 8001 점유 → 새 런처 시작 → 새 B 포트 못 바인딩 → 즉시 충돌 → PM2 재시작 → 무한루프 (1449회 재시작)

수정 전: PM2 → .venv/python.exe (런처, PID A) → 시스템 Python uvicorn (PID B, 고아) PM2가 A만 관리 → A 재시작 시 B 고아 남음 → 포트 충돌 → 재시작 루프
수정 후: PM2 → 시스템 Python 직접 (단일 PID) + PYTHONPATH=venv/Lib/site-packages 프로세스 1개, 고아 없음, 포트 충돌 없음, 재시작 0회

02
uvicorn 수동 taskkill
PASS
PM2가 직접 관리하는 시스템 Python 프로세스 종료 감지 → autorestart → 3초 후 재시작. 이전과 달리 고아 프로세스 없음 — 포트 충돌 없이 깔끔하게 재시작.
taskkill → PM2 감지 → 3초 후 시스템 Python uvicorn 단일 재시작 → 정상
03
포트 8001 다른 앱 점유
PASS (개선)
v1: 10회 → errored. 이제 50회 + min_uptime
uvicorn 포트 충돌 시 PM2 재시작. max_restarts=50, min_uptime=10s로 여유 생김. 점유 해제 후 정상 기동 시 카운터 리셋. 단, 영구 점유 시 50회 후 errored (개선 전 10회).
포트 점유 → 실패 → 점유 해제 후 성공 시 카운터 리셋 (min_uptime 10s)
04
절전 모드(Sleep) 후 복귀
PASS
PM2 데몬 생존. 이전에는 venv 런처(A) + 시스템 Python(B) 두 프로세스가 절전 중 꼬일 수 있었음. 이제 단일 프로세스라 복귀 후 안정적. 드물게 응답 없으면 PM2 autorestart 처리.
Sleep → 복귀 → 단일 프로세스 생존 → 즉시 정상 응답
05
화면 잠금(Win+L) 후 복귀
PASS
Lock Screen은 프로세스를 종료하지 않음. PM2와 단일 uvicorn 프로세스 모두 백그라운드에서 실행 유지. 복귀 즉시 서비스 정상.
잠금 → 프로세스 유지 → 잠금 해제 → 즉시 정상
06
SQLite DB 잠금 / 파일 오류
PASS
SQLite WAL 모드에서 쓰기 잠금은 해당 요청만 오류 반환, 프로세스 생존. uvicorn이 죽지 않으므로 PM2 재시작 없음. 잠금 해제 후 자동 정상화.
DB 잠금 → 해당 API만 503 → uvicorn 생존 → 잠금 해제 후 자동 복구
07
zombie-watchdog가 PM2 restart 트리거
PASS
10분마다 실행되는 zombie-watchdog가 daily-focus 비정상 감지 시 pm2 restart 호출. 이제 PM2 restart = 단일 시스템 Python 프로세스 재시작. 고아 없음, 터미널 창 없음.
zombie-watchdog → pm2 restart daily-focus → 단일 프로세스 깔끔 재시작
08
PYTHONPATH 패키지 오류 (uvicorn import 실패)
PASS (개선)
이제 PYTHONPATH=.venv/Lib/site-packages로 패키지 주입. 패키지 손상 시 uvicorn import 실패 → PM2 재시작. max_restarts=50으로 여유 있음. venv reinstall로 해결 가능.
패키지 손상 → import 실패 → PM2 50회 시도 → pip install -r requirements 후 정상
09
메모리 부족 (OOM)
WARN
uvicorn OOM 종료 → PM2 재시작. OOM 원인이 해소되면 정상 복구. OOM 지속 시 재시작 루프 → max_restarts 50회 후 errored. 이전 대비 이중 프로세스 없어 메모리 사용량 자체가 줄어 OOM 가능성 낮아짐.
OOM → 종료 → PM2 재시작 → 원인 해소 시 정상 → OOM 지속 시 50회 후 errored
10
pm2 stop daily-focus (의도적 중지)
FAIL (의도적)
pm2 stop은 정상 종료 → autorestart 비활성. 재시작하려면 pm2 start daily-focus. 이건 예상된 동작으로 버그 아님. 이전과 달리 stop 시 고아 시스템 Python이 포트 계속 점유하던 문제도 사라짐.
pm2 stop → 단일 프로세스 깔끔 종료 → pm2 start로 재개 가능
